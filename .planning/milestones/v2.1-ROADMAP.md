# Milestone v2.1: Multi-Provider LLM Integration (Azure OpenAI + Anthropic Claude via MGTI)

**Status:** SHIPPED 2026-05-22
**Phases:** 1-5
**Total Plans:** 20

## Overview

Added Anthropic Claude (via the MGTI Apigee/Bedrock proxy) as a session-selectable LLM provider alongside the existing Azure OpenAI integration. The five-phase structure followed a strict **seam → Azure extraction (parity-tested) → Anthropic adapter (no UI) → strict-tools + smoke test → UI toggle** sequence. The abstraction seam landed first against the existing OpenAI flow to prevent interface drift; a hard parity gate after Phase 2 confirmed byte-identical Azure behavior before Anthropic was introduced; the smoke test gated Phase 5 so users never see a provider toggle that hasn't been live-tested end-to-end. Existing Azure OpenAI flow remained functional and default-selected throughout — zero behavior change for existing users until they opt into Anthropic in Phase 5.

## Phases

### Phase 1: Abstraction Seam

**Goal**: Establish the provider-agnostic interface and supporting types in `src/llm/` so all subsequent provider work plugs into a stable seam. No call sites changed; no behavior change shipped.
**Depends on**: Nothing (first phase)
**Plans**: 3 plans

Plans:
- [x] 01-PLAN-01: Create src/llm/ package skeleton — `__init__.py`, `base.py` (`LLMClient` ABC), `errors.py` (6 typed errors), `types.py` (4 frozen+slots dataclasses)
- [x] 01-PLAN-02: Wire `LLMSettings` + `validate_config` (src/llm/config.py), `get_llm` factory with lazy-import registry (src/llm/__init__.py), adapter stubs, add `jsonschema` to requirements.txt
- [x] 01-PLAN-03: tests/test_llm_seam.py — 8-test pytest acceptance gate proving all 5 Phase 1 success criteria with zero live external dependencies

**Details:**

`src/llm/` package landed with `LLMClient` ABC enforcing the two-method contract (`complete`, `classify_with_tool`) at construction time. `get_llm(provider)` returns a cached `LLMClient` resolving the provider per the explicit-kwarg > session-state > env-default order; `LLM_PROVIDER_DEFAULT` falls back to `azure_openai`. `validate_config(provider)` raises `LLMConfigError` listing every missing variable. API keys never appear in any log line or `repr()` output across the package. Six typed errors form a flat hierarchy: `LLMError` + `LLMAuthError`, `LLMTransientError`, `LLMGuardrailError`, `LLMSchemaError`, `LLMTimeoutError`, `LLMConfigError`. Four frozen+slots dataclasses (`ToolSchema`, `ToolCall`, `ClassificationResultV1`, `IntentResult`) live at the adapter boundary; `ToolCall.raw_response` uses `field(repr=False)` so debug payloads never leak into logs.

### Phase 3: Anthropic MGTI Adapter

**Goal**: Add `AnthropicMGTIClient` with text-mode `complete()` against the MGTI Apigee proxy, full typed-error mapping from MGTI HTTP responses, and observability hooks. Adapter reachable via `get_llm("anthropic_mgti")` but no UI exposes it yet.
**Depends on**: Phase 2 (parity gate must pass before introducing Anthropic)
**Plans**: 4 plans

Plans:
- [x] 03-PLAN-01: Extend .env.example with 9 Phase 3 variables (LLM_PROVIDER_DEFAULT + 8 ANTHROPIC_*); add `llm_provider_loaded` startup log to AzureOpenAIClient.__init__ (CFG-02, CFG-04, OBS-01, OBS-04)
- [x] 03-PLAN-02: Add per-provider dispatch on `e.provider` in `src/llm/_compat.py` for LLMAuthError, LLMTimeoutError, LLMTransientError, catch-all LLMError — pays down Phase 2 known debt (ERR-02, ERR-03)
- [x] 03-PLAN-03: Replace Phase 1 stub with full `AnthropicMGTIClient`: model-prefix validation + startup log, complete() with MGTI HTTP + correlation UUID + structured log + guardrail-before-emptiness response handling (ADP-03..06, ADP-08, ERR-02, ERR-03, OBS-01, OBS-04, TOOL-07)
- [x] 03-PLAN-04: tests/test_phase3_adapter.py — 21-test acceptance gate proving all 5 Phase 3 success criteria + end-to-end compat dispatch

**Details:**

`get_llm("anthropic_mgti").complete(...)` performs a `POST {base_url}/model/{model}/messages` with `X-Api-Key`, `Content-Type: application/json`, and a fresh `X-Correlation-Id` UUID per call. Constructing `AnthropicMGTIClient` with a model name not starting `eu.anthropic.claude-` raises `LLMConfigError`; constructing with an `eu.anthropic.claude-opus-4-7*` model results in `temperature`/`top_p`/`top_k` being omitted from the request body. MGTI HTTP 401/403 → `LLMAuthError`; 429/5xx → `LLMTransientError`; `requests.Timeout` → `LLMTimeoutError`; `stop_reason == "guardrail_intervened"` → `LLMGuardrailError` (guardrail check BEFORE content-emptiness check — Pitfall 4 lock). Field-name normalization at log boundary: Anthropic's `usage.input_tokens`/`output_tokens` → `llm_prompt_tokens`/`llm_completion_tokens` so dashboards aggregate across providers without per-provider field maps. `_compat.py` dispatches on `e.provider` in 4 branches — Anthropic timeouts no longer surface as "Azure OpenAI API call failed".

### Phase 4: Strict-Tools + Smoke Test

**Goal**: Wire Anthropic strict-tools for intent classification (single Python source of truth, `chart_requested`/`chart_type` stay out of the LLM schema) and ship `scripts/smoke_llm.py` as the live-credential gate that must pass for both providers before Phase 5.
**Depends on**: Phase 3
**Plans**: 4 plans

Plans:
- [x] 04-PLAN-01: Derive INTENT_TOOL from ClassificationResultV1 (single source of truth, TOOL-02); migrate `classify_intent` from `complete()`+json.loads to `classify_with_tool()` with heuristic-merge AFTER LLM result (TOOL-04)
- [x] 04-PLAN-02: Replace `AnthropicMGTIClient.classify_with_tool` stub with strict-tools path (tools + tool_choice + jsonschema.validate); add env-flag-gated `_classify_via_text_mode` escape hatch; shared `_post_messages` helper; tools_supported + llm_tool_mode log fields (ADP-07, ADP-09, TOOL-05, TOOL-06)
- [x] 04-PLAN-03: Create `scripts/smoke_llm.py` operator-run live-credential gate: 3 checks for Anthropic (service-info GET + complete + classify_with_tool), 2 for Azure; CONTINUE-ON-FAILURE; missing-cred SKIP/FAIL matrix (SMK-01..05)
- [x] 04-PLAN-04: tests/test_phase4_strict_tools.py — 30-test acceptance gate proving all 5 Phase 4 success criteria + 9 error-matrix rows + COMPAT-DISPATCH pair (LLMSchemaError, LLMGuardrailError)

**Details:**

`INTENT_TOOL` is derived programmatically from `ClassificationResultV1` via `typing.get_type_hints()` (single source of truth — TOOL-02); the derived `input_schema` does NOT contain `chart_requested` or `chart_type` fields (TOOL-03). `AnthropicMGTIClient.classify_with_tool(...)` sends `tools=[INTENT_TOOL]` and `tool_choice={"type": "tool", "name": ..., "disable_parallel_tool_use": True}`; on success extracts the `tool_use` block input and returns a `ToolCall` validated via `jsonschema.validate`; missing tool_use block raises `LLMSchemaError`. With `ANTHROPIC_TOOLS_SUPPORTED=false`, falls back transparently to text-mode + JSON parsing (env-flag-only fallback — no runtime auto-fallback for loud signal if proxy regresses). `classify_intent` merges the heuristic-populated `chart_requested`/`chart_type` AFTER the LLM result — LLM cannot overwrite the heuristic for these two fields. `scripts/smoke_llm.py` exists with `--provider azure_openai|anthropic_mgti|both`, exercises `complete()` + `classify_with_tool()` + Anthropic service-info diagnostic, exits non-zero on failure.

- 3 days from start to ship (2026-05-19 → 2026-05-22)

**Git range:** `f3cc42e feat(01-01)` → `cd07cae docs(05)`

**Key Decisions:**

- Provider abstraction layer (`abc.ABC` + two adapters) over per-call `if` branches — interface drift was the dominant pitfall to prevent
- Parity-first refactor — Azure extraction produced byte-identical output to today before Anthropic was introduced
- Anthropic adapter speaks native MGTI shape — no OpenAI-translation middleware
- Strict-tools mode for `classify_intent` only — `chart_requested`/`chart_type` stay out of the LLM schema (heuristic-populated)
- Default provider stays `azure_openai` — upgrade is byte-identical for existing deployments
- Single cache layer: `@st.cache_resource` is the only cache (Phase 1 module-level `_cache: dict` deleted in Phase 5)
- `_log_llm_call` intentionally duplicated between azure_openai.py and anthropic_mgti.py — no premature extraction until a third adapter exists
- `_compat.py` dispatches on `e.provider` for per-provider QueryError wording — prevents wrong-product-label UX bug
- `_render_provenance_caption` must never read `st.session_state` — historical messages survive provider switches with original provenance preserved (AST-based regression test locks the invariant at executable level)
- INTENT_TOOL derived from ClassificationResultV1 via `typing.get_type_hints()` — `fields().type` returns strings under `from __future__ import annotations` (RESEARCH.md Phase 4 Pitfall 1 guard)

**Issues Resolved:**

- Phase 2 known debt (LLMAuthError hardcoded Azure remediation) resolved by Phase 3's per-provider dispatch in `_compat.py`
- Anthropic guardrail interventions (HTTP 200 + empty content) correctly raise `LLMGuardrailError` not silent-success
- Mid-session provider switch no longer breaks historical-message provenance (Pitfall 11 lock)
- `st.chat_input` disables on missing-env warning — blocks unrecoverable queries before they hit the LLM

**Issues Deferred:**

- **SMK-05 live smoke run against staging gateway** — pre-authorized operator gate (`python scripts/smoke_llm.py --provider both --verbose`). Smoke script artifact exists and is structurally verified; live execution requires staging credentials not present in CI/dev. Must run before any production deploy. See audit report `.planning/milestones/v2.1-MILESTONE-AUDIT.md` §7.

**Technical Debt Incurred:**

- Duplicated `_log_llm_call` helper between `src/llm/azure_openai.py:52` and `src/llm/anthropic_mgti.py:49` — locked Phase 2 decision; both files annotate the duplication. Extract when a third adapter is added (out of scope for v2.1).
