# Phase 9, Plan 01 — Altair `loro_piana` theme module

## Objective

Create `src/ui/altair_theme.py` — a side-effect module whose import registers and activates the `loro_piana` Altair theme for SNOWGREP v2.2. This is the foundation for Phase 9's chart restyle (Plan 03) and for theme activation in `app.py` (Plan 04). Maps to **DVZ-04**: every Altair chart in the app must render with cashmere-chrome (EB Garamond title, Inter labels, warm-beige gridlines, no axis box) while data marks use the vibrant categorical palette. Using Altair 6's new `@alt.theme.register` decorator API (NOT the deprecated `alt.themes.*` API) registers the theme at import-time exactly once per Python process; subsequent `st.altair_chart()` calls bake the theme config into chart spec JSON automatically.

## Context

- Authoritative spec: `.planning/phases/09-data-visualization/09-CONTEXT.md` — section "Chart palette + chart-type logic". User-approved deviation from REQUIREMENTS.md DVZ-04 literal cashmere palette: chart **data** is vibrant categorical; cashmere applies to chart **chrome** only.
- Research: `.planning/phases/09-data-visualization/09-RESEARCH.md` — Pattern 1 verifies Altair 6.0.0 + `alt.theme.register` decorator works, and Pitfall 1 documents why the old `alt.themes.*` API must be avoided (emits `AltairDeprecationWarning`).
- Vibrant palette colors: `#C0392B` (crimson), `#2E5BBA` (royal blue), `#2E7D32` (forest green), `#E67E22` (burnt orange), `#F39C12` (mustard yellow). Order matters — `palette[0]` is the single-series line-chart color.
- Phase 6 single-CSS-source pattern: this is the canonical chart palette source; future modules import from here rather than redefine.
- Tasks 1 ships a single atomic commit.
